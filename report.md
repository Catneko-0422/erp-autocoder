## 01 研究背景與動機

### 問題描述

企業在導入 ERP 系統時，每個料件（電子零件、PCB、電容器等）需要一組唯一且具規則性的 16 碼料號（PART NO.）。目前多數企業仍依賴人工查表編碼，存在以下問題：

- **人工失誤率高**：不同操作員對同一規格可能給出不同編碼
- **效率低落**：查閱編碼規則紙本文件，每筆料號需耗時數分鐘
- **知識斷層**：編碼規則僅存在資深員工腦中，新人培訓成本高
- **缺乏稽核**：無法追溯料號是誰、何時、依據哪個版本規則產生

### 研究動機

- **自動化編碼可大幅降低人為錯誤**，提升料號一致性
- **保留編碼規則的版本管理**，適應企業逐年調整的編碼原則
- **引入 LLM 作為輔助**，讓使用者以自然語言描述規格即可自動產生料號
- 解決效益：預估減少 90% 人工編碼時間，錯誤率趨近於零

## 02 研究方法

### 技術路線

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 規則樹設計 │ → │ 前後端開發 │ → │ LLM 整合 │ → │ 評估測試 │
│ (Adj List)│   │Flask+React│   │RAG+Groq  │   │ 準確率   │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### 使用技術 / 框架

| 類別 | 技術 |
|---|---|
| 後端框架 | Flask 3 + Python 3.14 |
| 前端框架 | React 19 + TypeScript + Vite 8 |
| 資料庫 | PostgreSQL 16 + Redis 7 |
| LLM API | Groq API (llama-3.1-8b-instant) |
| RAG 管道 | LangChain + 向量檢索 |
| 容器化 | Docker Compose |
| 認證機制 | JWT + Argon2 + RBAC |

### 模型選擇

選用 **llama-3.1-8b-instant**（透過 Groq API 调用）：

- 8B 參數規模可在單張 GPU 或純 CPU 推論，適合產學合作場景
- 支援工具呼叫（function calling），可與編碼規則 API 整合
- Groq API 提供極低延遲推論（< 200ms），符合即時編碼需求

### 架構設計

```
使用者描述 → LLM (Groq) → 結構化欄位 → 規則樹驗證 → PART NO.
                      ↕ (RAG 檢索)
                  編碼規則向量庫
```

## 07 結論與下一步

### 主要發現

- 以 Adjacency List 儲存的規則樹結構可靈活支援多版本編碼原則，節點的 `field_type`（options/option/input/fixed）設計能涵蓋絕大多數編碼場景
- 兩層式物料選擇（群組 → 具體物料）有效簡化使用者操作流程
- 前端的樂觀更新大幅提升規則樹編輯的流暢度

### 技術亮點

- 規則樹支援四種欄位類型（選擇集、選項、輸入、固定值）
- 角色化權限設計讓規則制定者（RuleMaker）與一般使用者（User）職責分離
- Docker 容器化確保開發與部署環境一致

### 下一步計畫

1. 完成 LLM / Groq API 整合：使用者輸入自然語言描述，系統自動填入對應欄位並產生料號
2. 導入 RAG 機制：將歷史編碼記錄作為檢索庫，提升 LLM 生成準確率
3. 擴充更多材料類別：電阻、IC、二極體等完整電子料件編碼規則
4. 效能最佳化與使用者測試

### 遇到的困難

- 規則樹的巢狀結構在前端 render 效能上的挑戰（透過虛擬化與樂觀更新解決）
- LLM 對中文技術規格的語意理解不夠精確（需透過 few-shot prompting 改善）
- Docker 環境中 CORS 與反代配置的相容性問題

### 學習心得

- 深入理解了 Adjacency List 與巢狀樹狀結構在關聯式資料庫中的設計與操作
- 掌握了 React 19 搭配 TypeScript 的完整前後端分離開發流程
- 體驗了從系統設計、開發、容器化到部署的全端專案實作經驗
- 認識到 LLM 在垂直領域應用中，RAG 與 prompt engineering 的重要性
